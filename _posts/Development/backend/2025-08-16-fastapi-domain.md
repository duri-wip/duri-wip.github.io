# domain 계층의 구현

데이터 모델을 만들때 이 모델에 따라서 데이터 베이스도 디자인해야되는건 아니다.

- user라는 테이블에 모든 데이터를 포함하고, 데이터 베이스를 만들때 이렇게 분리해서 만들 수 있다.
  profile 모델을 보면 id 속성이 없는데, 이처럼 데이터만 가지고 있는 도메인 객체를 값객체(vo)라고 한다..
  ???
  도메인 주도 설계에서는 도메인을 핵심 도메인과 서브 도메인으로 나눈다.

# 인터페이스 의 구현

그리고 유저 도메인을 영속화하기 위한 모듈을 구현한다.
이름에 i를 붙이는 이유는 명시적으로 인터페이스임을 나타내기 위함이다. 이 인터페이스 실제 구현체는 인프라 계층에 존재한다. 도메인 계층에존재하는 이 모듈은 어느 계층에서나 사용할 수 있으며, 인프라 계층보다 고 수준의 계층에서 사용하 ㄹ떄는 의존성이 역전돼있다.

유저 저장을 구현할때 순서가 이렇다.
먼저 실제 서비스가 구현되는 로직을 만들어본다
애플리케이션 계층에 유저 저장하는 로직을 만들고 -> 여기에 해당하는 저수준의 구현체를 구현한다.
예를 들어서
application/user_service에서 create_user기능을 구현한다.
이때 이 계층은 고수준 계층에서 구현하는 구현체이므로 저수준의 구현체와 소통이 가능하다
그래서 유저를 생성할 때 데이터베이스에 저장이 되어야하는 부분은
