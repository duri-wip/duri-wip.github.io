**Heartbeat 기반 솔루션**은 고가용성을 유지하기 위한 클러스터 관리 방법 중 하나로, 여러 서버 또는 노드의 상태를 실시간으로 모니터링하고, 장애가 발생하면 즉시 대응할 수 있도록 설계된 시스템입니다. 이러한 시스템은 특히 **Failover**(장애 조치)를 자동으로 처리하는 데 중점을 두며, 보통 **Pacemaker**와 같은 고가용성 클러스터 관리 도구와 함께 사용됩니다. 

### 1. **Heartbeat란?**
**Heartbeat**는 클러스터 내의 서버 간에 정기적인 신호(heartbeat)를 주고받으며 서로의 상태를 확인하는 프로토콜입니다. 이 신호가 중단되면 노드 간의 통신이 끊긴 것으로 간주되고, 이를 기반으로 Failover가 트리거됩니다. 주로 두 가지 주요 역할을 합니다:
   - **서버 모니터링**: 클러스터 내의 각 서버가 정상적으로 동작하는지 주기적으로 확인합니다.
   - **Failover 트리거**: 한 서버가 응답하지 않을 경우, 다른 서버가 이를 감지하고 장애 복구 작업을 시작합니다.

Heartbeat는 **Pacemaker**와 같은 클러스터 관리 시스템에 의해 관리되며, 클러스터 노드들이 주기적으로 상태 정보를 교환함으로써 서로의 상태를 파악할 수 있게 해줍니다.

---

### 2. **Heartbeat 기반 솔루션의 핵심 구성 요소**
Heartbeat 기반 솔루션을 구성하는 주요 요소들은 다음과 같습니다:

#### (1) **Pacemaker**
Pacemaker는 고가용성 클러스터를 관리하는 데 사용되는 오픈소스 도구로, 장애 감지 및 Failover를 처리합니다. Pacemaker는 클러스터의 리소스(서비스, 애플리케이션, 데이터베이스 등)를 관리하며, **Corosync**나 **Heartbeat**와 같은 통신 레이어를 통해 각 노드의 상태를 모니터링합니다.

Pacemaker의 주요 기능은 다음과 같습니다:
- **Failover 관리**: 마스터가 장애를 일으키면 Pacemaker가 슬레이브를 승격하여 새로운 마스터로 전환하는 작업을 수행합니다.
- **클러스터 리소스 관리**: 애플리케이션, 데이터베이스, 파일시스템 등 클러스터에서 관리해야 할 다양한 리소스를 모니터링하고 관리합니다.
- **멀티 노드 지원**: Pacemaker는 여러 노드(서버)를 포함하는 클러스터를 지원하며, 노드 간의 고가용성 관리와 장애 복구를 제공합니다.

#### (2) **Corosync**
Corosync는 Pacemaker와 함께 사용되는 클러스터 통신 및 상태 관리 도구입니다. 클러스터 내의 각 노드가 상태 정보를 교환하는 역할을 하며, 클러스터 내에서 장애 발생 시 이를 감지하고 Pacemaker에게 알립니다.

Corosync의 기능은 다음과 같습니다:
- **클러스터 상태 유지**: 노드 간 통신을 관리하고, 각 노드의 상태를 주기적으로 보고합니다.
- **Heartbeat 구현**: Corosync는 클러스터 내 노드들이 정기적으로 Heartbeat 신호를 주고받으며, 이를 통해 장애를 감지합니다.
- **멀티캐스트/유니캐스트 통신**: Corosync는 멀티캐스트 또는 유니캐스트 네트워크에서 사용할 수 있습니다.

---

### 3. **Heartbeat 기반 Failover 동작 원리**
Heartbeat 기반 솔루션의 동작 원리는 다음과 같습니다:

1. **노드 간 Heartbeat 신호 주고받기**: 
   클러스터 내의 노드들이 주기적으로 서로 Heartbeat 신호를 주고받습니다. 이를 통해 각 노드가 정상적으로 작동하는지 확인합니다.

2. **장애 감지**:
   만약 특정 노드가 Heartbeat 신호에 응답하지 않으면, 해당 노드가 장애를 일으켰다고 간주합니다. 장애가 발생한 노드는 더 이상 클러스터의 일부로 인식되지 않고, Pacemaker는 이 상태를 감지하여 Failover 프로세스를 시작합니다.

3. **자원 이동 및 Failover**:
   장애가 감지되면, Pacemaker는 장애가 발생한 노드에서 실행 중인 리소스(예: 데이터베이스, 서비스)를 다른 노드로 이동시킵니다. 예를 들어, 마스터 데이터베이스가 장애를 겪었다면, Pacemaker는 슬레이브 데이터베이스를 새로운 마스터로 승격하고, 클러스터 리소스를 재배치합니다.

4. **장애 복구**:
   장애 복구가 완료되면, 새로운 마스터가 정상적으로 작동을 시작하고, 클러스터의 나머지 노드와 동기화됩니다. 장애가 해결된 후에도 Pacemaker는 지속적으로 클러스터의 상태를 모니터링합니다.

### 4. **Pacemaker와 Heartbeat 설정 예시**
Pacemaker와 Heartbeat를 이용한 고가용성 PostgreSQL 클러스터 설정의 기본 흐름은 다음과 같습니다:

#### 1) **환경 설정**
   - **Pacemaker와 Corosync 설치**: 고가용성 클러스터의 기본 소프트웨어를 설치합니다. `apt` 또는 `yum`을 사용해 Pacemaker와 Corosync를 설치할 수 있습니다.
   
     ```bash
     sudo apt-get install pacemaker corosync
     ```

#### 2) **Corosync 설정** 
   - Corosync 설정 파일을 편집하여 클러스터 노드 간의 통신 방법을 정의합니다.
   
     `/etc/corosync/corosync.conf` 파일을 수정하여 클러스터 내의 노드를 정의하고, 유니캐스트 또는 멀티캐스트 통신을 설정합니다.

     ```bash
     totem {
       version: 2
       secauth: off
       threads: 0
       interface {
         ringnumber: 0
         bindnetaddr: 192.168.1.0
         mcastaddr: 226.94.1.1
         mcastport: 5405
       }
     }
     ```

#### 3) **Pacemaker 설정**
   - Pacemaker에서 PostgreSQL 또는 다른 리소스를 관리하도록 리소스를 정의합니다.
   
     ```bash
     sudo crm configure primitive pgsqld ocf:heartbeat:pgsql \
     op start timeout=60s interval=0 \
     op stop timeout=60s interval=0 \
     op monitor timeout=60s interval=10s \
     params pgctl="/usr/pgsql/bin/pg_ctl"
     ```

#### 4) **Failover 설정 및 테스트**
   - Pacemaker를 통해 고가용성 클러스터에서 장애가 발생했을 때, Failover가 제대로 작동하는지 테스트합니다. 장애 시나리오를 시뮬레이션하고, 마스터 장애 발생 시 슬레이브가 마스터로 정상적으로 승격되는지 확인합니다.

### 5. **Heartbeat 기반 솔루션의 장점**
1. **빠른 장애 감지와 복구**:
   - Heartbeat를 통해 장애를 실시간으로 감지하고, 빠르게 복구 작업을 수행할 수 있습니다. 이는 서비스 중단 시간을 최소화하는 데 매우 유리합니다.

2. **클러스터 전반 관리**:
   - Pacemaker와 함께 사용할 경우, 고가용성 리소스 관리와 클러스터 전체의 상태를 모니터링할 수 있습니다. 이는 복잡한 서비스 아키텍처에서 효율적인 관리가 가능합니다.

3. **유연한 확장성**:
   - 클러스터에 추가적인 노드를 쉽게 추가할 수 있으며, 노드 간 통신 설정을 변경하여 확장성을 높일 수 있습니다.

4. **자동 Failover 지원**:
   - Heartbeat 기반 솔루션은 장애 발생 시 자동으로 장애 복구를 처리하고, 클러스터 내에서 리소스를 재배치할 수 있습니다.

---

### 6. **Heartbeat 기반 솔루션의 단점**
1. **잘못된 장애 감지 (Split-Brain 문제)**:
   - 네트워크 문제가 발생하면 클러스터 내의 일부 노드들이 서로를 잘못된 상태로 판단하여 각각 마스터 역할을 수행하려는 상황이 발생할 수 있습니다. 이를 **Split-Brain**이라고 하며, 데이터 일관성 문제가 발생할 수 있습니다.
   
   이를 방지하기 위해 **STONITH (Shoot The Other Node In The Head)**라는 강제 종료 메커니즘을 사용하여 한 노드를 완전히 종료시켜 문제를 해결할 수 있습니다.

2. **복잡한 설정**:
   - Pacemaker와 Corosync를 설정하고, 장애 시 Failover를 완벽하게 관리하려면 복잡한 설정이 필요할 수 있습니다.

### 결론:
Heartbeat 기반 솔루션은 고가용성을 위해 매우 효과적인 방식이며, Pacemaker 및 Corosync와 함께 사용하면 장애 발생 시 빠른 복구와 안정적인 클러스터 운영이 가능합니다. MLOps 프로젝트에서 데이터베이스와 모델 서버의 고가용성을 위해 Heartbeat 기반 솔루션을 활용하면 안정적인 시스템 운영이 가능합니다.
